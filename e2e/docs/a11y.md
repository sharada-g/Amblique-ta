# Accessibility Testing

Full-page accessibility scanning for critical storefront pages using [axe-core](https://github.com/dequelabs/axe-core) via `@axe-core/playwright`. Tests run in a real browser against the running storefront, catching issues that component-level Storybook a11y tests cannot detect (page composition, routing, layout, responsive behavior).

## Overview

- **Scope**: WCAG 2.1 AA (`wcag2a`, `wcag2aa`, `wcag21aa` tags)
- **Viewports**: Desktop (1200×900) and mobile (Pixel 7 emulation) — `pnpm a11y` runs both passes sequentially via `scripts/run-a11y.ts` (desktop first, then `PLAYWRIGHT_MOBILE=true` for mobile)
- **Baseline**: Violation counts are snapshotted per page per viewport. CI fails only when **critical** or **serious** violations increase or a new critical/serious rule appears; moderate/minor regressions are logged as informational but do not block CI.
- **Retries**: Infrastructure failures (timeouts, navigation errors) are retried up to 2 times via `.retry(2)` on the Feature. Confirmed a11y regressions (thrown as `A11yBaselineError`) are never retried — the `a11yNoRetry` plugin suppresses retries for those, since axe scans are deterministic and re-running a real violation always reproduces it.

## Quick Start

```bash
# Run a11y scans against the running storefront
pnpm a11y

# Generate a Markdown report with ticket-friendly violation details
pnpm a11y:report

# Scan all pages and write current counts as the new baseline
pnpm a11y:update-baseline
```

The storefront dev server must be running before executing tests. Start it with:

```bash
# From packages/template-retail-rsc-app
pnpm dev
```

## Console Output

Each scan prints a clear banner identifying the page, viewport, and WCAG standard:

```
════════════════════════════════════════
  A11Y SCAN: homepage | desktop
  Standard: WCAG 2.1 AA (wcag2a, wcag2aa, wcag21aa)
════════════════════════════════════════
```

The severity legend is printed once at the start of the suite run. On completion each scan prints one of:

- `✓ PASS: homepage/desktop — 5 violations (within baseline)` — scan passed
- `↓ homepage/desktop — violations decreased, run pnpm a11y:update-baseline` — improvement detected
- `  ⓘ 2 moderate/minor rules exceeded baseline (not blocking — update with pnpm a11y:update-baseline)` — moderate or minor counts increased; informational only, printed below the PASS line
- Failure: full violation report followed by a summary of new/increased critical/serious rules

## Generating a Markdown Report

`pnpm a11y:report` runs all scans with result collection enabled and writes two files to `a11y-report/`:

- **`report.md`** — Markdown summary table (page × viewport × severity counts) plus per-violation detail with rule ID, description, axe-core help URL, and HTML snippets of affected elements — ready for filing as tickets
- **`report.html`** — styled HTML version of the same report, intended for CI artifact upload and browser viewing

`a11y-report/` is gitignored and treated as a generated artifact. Generate it locally when needed:

```bash
pnpm a11y:report
# Writes a11y-report/report.md and a11y-report/report.html
```

## How the Baseline Works

The baseline is stored in `a11y-baseline.json` at the package root. Each entry maps a `<pageKey>/<viewport>` key to a `Record<ruleId, nodeCount>`:

```json
{
  "homepage/desktop": { "color-contrast": 3, "image-alt": 1 },
  "homepage/mobile":  { "color-contrast": 2 },
  "login/desktop":    {}
}
```

**On every test run** the spec compares the current axe output against the baseline:
- **New critical/serious rule** or **more critical/serious violations** than baseline → test **fails**
- **New moderate/minor rule** or **more moderate/minor violations** than baseline → logged as informational, test **passes**
- **Fewer violations** than baseline → test **passes** with a log message suggesting a baseline update

**Ratchet-down workflow**: Once you fix a known issue, update the baseline to lock in the improvement:

```bash
pnpm a11y:update-baseline
# Review the diff in a11y-baseline.json, then commit it
git add a11y-baseline.json
git commit -m "chore: ratchet down a11y baseline after fixing color-contrast on homepage"
```

**Initial setup for new pages**: When you add a new page, run `pnpm a11y:update-baseline` — it creates the baseline entries automatically and populates them with current violation counts. Commit the result.

## Running at Specific Viewports

`pnpm a11y` always runs both passes. To target a single viewport, invoke the test runner directly:

```bash
# Desktop only
tsx src/scripts/cli/test-runner.ts --grep @a11y

# Mobile only (Pixel 7 emulation)
PLAYWRIGHT_MOBILE=true tsx src/scripts/cli/test-runner.ts --grep @a11y
```

> **Note:** `pnpm e2e run --multiple desktop --grep "@a11y"` does **not** work — CodeceptJS `run-multiple` does not forward `--grep` to its workers, so all tests run instead of just the a11y suite.

## Adding a New Page

A11y specs live in `src/specs/core/a11y/` and are split by authentication requirement:

| File | Auth | Use case |
|------|------|----------|
| `public.spec.ts` | None | Public pages (homepage, PLP, PDP, search, cart, checkout, login, signup) |
| `account.spec.ts` | Fresh signup | Account pages (details, addresses, payment methods, wishlist) |
| `orders.spec.ts` | Pre-existing user | Order pages (order list, order details) — requires `E2E_TEST_USER_EMAIL` / `E2E_TEST_USER_PASSWORD` env vars; scenarios are skipped when credentials are not set |

1. Add a `Scenario` to the appropriate file following the existing pattern:

```typescript
Scenario('My New Page accessibility', async () => {
    const viewport = await beginScan('my-new-page');
    myPage.navigate();
    await scanAndAssert('my-new-page', viewport);
}).tag('@my-new-page');
```

2. Run `pnpm a11y:update-baseline` — it creates the baseline entries for the new page automatically. Review the diff in `a11y-baseline.json`, then commit it.

## Severity Levels

axe-core reports violations with four impact levels:

| Level      | Meaning                                               |
|------------|-------------------------------------------------------|
| `critical` | Blocks access entirely for users with disabilities    |
| `serious`  | Creates significant barriers; should be fixed quickly |
| `moderate` | Creates difficulty but workarounds may exist          |
| `minor`    | Low impact; fix when convenient                       |

Severity appears in several places in test output:
- **Scan banner**: printed before each scan alongside the WCAG standard
- **Failure summary**: each failing rule is shown inline as `rule-id [impact]: N violations` so you can assess priority at a glance
- **Full violation report**: printed above the summary, grouped by impact level with rule descriptions, help URLs, and affected element selectors/HTML snippets
- **Markdown report**: summary table + per-rule detail sections with help URLs and HTML snippets for ticket creation

## Environment Variables

| Variable | Values | Description |
|---|---|---|
| `BASE_URL` | URL | Storefront URL to scan. Defaults to `http://localhost:5173`. |
| `PLAYWRIGHT_MOBILE` | `true` | Enable Pixel 7 emulation. Set automatically by `pnpm a11y` for the mobile pass; use directly when running a single mobile pass. |
| `A11Y_UPDATE_BASELINE` | `true` | Write current violation counts to `a11y-baseline.json` (package root) instead of asserting. Set automatically by `pnpm a11y:update-baseline`. |
| `A11Y_COLLECT_RESULTS` | `true` | Write scan results to `a11y-report/data/` for offline report generation. Set automatically by `pnpm a11y:report`. |

## CI Integration

A11y scans run as a parallel job alongside the core E2E suite on every PR, merge group event, and push to `main` or `release-*` branches. Both jobs share the same MRT deployment, so there is no extra deploy cost.

**Caller workflow**: `.github/workflows/e2e-core-pr.yml`
**Reusable runner**: `.github/workflows/e2e-pr-runner.yml` — the `run_a11y` job defined here handles the scan, reporting, and artifact upload.

**Kill switch**: The `run_a11y` job is gated on the `ENABLE_A11Y_CI` repository variable. Set it to `false` to disable a11y CI globally without modifying any workflow file.

**Command**: `pnpm a11y:report --no-ai` — runs scans in collect mode and generates both `a11y-report/report.md` and `a11y-report/report.html`.

**Job summary**: `a11y-report/report.md` is appended to the GitHub Actions job summary after each run, so violation details are visible directly in the Actions UI without downloading anything.

**HTML report artifact**: `a11y-report/report.html` is uploaded as a CI artifact (`a11y-report-<run-id>`) after each run. Download it from the Actions run summary to view violation details in a browser without cloning the branch.
